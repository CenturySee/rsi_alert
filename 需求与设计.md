# RSI 预警页面 — 需求与设计文档

## 一、需求描述

做一个本地 Web 页面，展示指定品种的日线 K 线与 RSI 指标，并计算**下个交易日 RSI 触及各阈值时对应的价格和涨跌幅**。

### 功能点

1. **K 线图**：展示某品种最近 300 个交易日的日线数据（OHLCV）
2. **RSI 指标图**：RSI 周期参数可自定义（默认 6）
3. **预警价格表**：计算下一交易日收盘价需达到多少，才能使 RSI 恰好触及以下阈值：
   - 下穿：30 / 25 / 20 / 15 / 10（可自定义）
   - 上穿：75 / 80（可自定义）
   - 同时显示该价格相对当日收盘价的涨跌幅
4. **品种名称**：可自定义输入
5. **阈值**：支持增删，下穿/上穿分开配置

**展示逻辑：**
- **状态条**（图上方）：突出展示「当前 RSI 值 + 距最近阈值还需的涨跌幅 + 下一交易日日期」，这是用户最先要看的信息
- **触发价上图**：K 线图上用水平 `markLine` 把各触发价画成横线（下穿绿 / 上穿红），不要只放在表格里
- **预警表三种状态**：
  - 正常：`¥价格  ±x%`
  - 已触及（次日收盘不变即超过阈值）：标灰 + `已触及 ✓`，不显示价格
  - 超涨跌停（解出的价格越过当日涨跌停板）：标注 `需超涨跌停` 并置灰，**不显示一个假的不可达价格**
- **下一交易日日期**：预警针对下个交易日，表头与状态条都标注具体日期（如 `06-24`），跨周/节假日按交易日历顺延

**交互说明：**
- 顶部控制栏：品种代码 + 市场（默认「自动」按代码前缀推断 SH/SZ/BJ，可手动覆盖）+ RSI 周期 + 阈值弹窗 + 查询 + 强制更新
- **查询 vs 强制更新分开**：
  - 「查询」/ 代码框回车 → 普通查询（命中缓存则走缓存）
  - 「⟳强制更新」→ 调 `/api/refresh` 忽略缓存重拉 opentdx
- **图表联动**：K 线图与 RSI 图 X 轴 / 十字线联动（`echarts.connect`），共用同一 `dataZoom`；hover 预警表某行时高亮 K 线图上对应的触发价横线
- **参数持久化**：RSI 周期、下穿/上穿阈值列表用 `localStorage` 保存，下次打开沿用
- 阈值设置弹窗：可分别增删下穿阈值列表和上穿阈值列表
- 预警表中涨用红色、跌用绿色（A 股配色）

## 五、核心算法

### RSI 计算（通达信 SMA / Wilder 口径）

以 `C:\\xzq\\projects\\quant\\hongli\\scripts\\calc_rsi_trigger_price.py` 中的 **`calc_rsi`** 函数为准
（基于 `tdx_sma`，即通达信 `SMA(N,1)` 递推平滑，等价于 Wilder 平滑，有记忆性）。

> ⚠️ **不要用 `calc_rsi2`**：那是无平滑的累加式（Cutler's RSI，`sum(ups)/(sum(ups)-sum(downs))`），
> 与通达信 RSI(6) 数值会明显不同，会导致第八节「与通达信对比」验证对不上。
> `calc_trigger_price` 里 `calc_rsi2` 那行是被注释掉的，保持用 `calc_rsi`。

### 预警价格反推（解析解，替代暴力扫描）

原脚本 `calc_trigger_price` 用 0.001 步长从 `max_price` 扫到 `min_price`、每步整段重算 RSI（O(n)），
慢且有精度抖动。**改用闭式求解**，因为下一根的 RSI 只取决于上一根的递推状态 `avg_gain_{t-1}`、`avg_abs_{t-1}`：

- 先对历史序列**预计算一次** `avg_gain_{t-1}`、`avg_abs_{t-1}`（通达信 SMA 递推到最后一根）。
- 设次日收盘相对今收的变动为 `delta`：
  - 上涨（`delta > 0`）：`gain = delta`，`abs = delta`
  - 下跌（`delta < 0`）：`gain = 0`，`abs = -delta`
- 平滑递推（`N` = 周期，weight=1）：
  `avg_gain_t = (gain + (N-1)*avg_gain_{t-1}) / N`，`avg_abs_t = (abs + (N-1)*avg_abs_{t-1}) / N`
  目标 `RSI = avg_gain_t / avg_abs_t * 100 = 阈值`。
- 两个分支对 `delta` 都是线性的 → 直接解出 `delta`，再得 `target_price = close_today + delta`，O(1) 且精确。
- 算完用 `calc_rsi` 代回验证一次（应等于阈值）。

> 注意当前日期与 K 线数据中的日期不要重合（沿用脚本盘中去掉实时最新一根的 `remove_latest` 处理）。

**涨跌停区间约束（按市场/板块）**：原脚本硬编码 `±10%`，覆盖不了科创/创业板 `±20%`、北交所 `±30%`、ST `±5%`。
按代码前缀判断对应涨跌幅上限来约束反推价格；若解出的 `target_price` 超出当日涨跌停，标注「需超涨跌停」。（期货 v1 不支持，见第十二节）

**涨跌幅：**

```
pct_change = (target_price - close_today) / close_today * 100
```

> **注意**：若解出的 `delta`（loss_needed / gain_needed）符号反了，说明当前 avg_gain/avg_abs 状态下，
> 即使次日收盘不变也已经超过该阈值，此时标注"已触及"。

## 六、前端实现要点

- 框架：纯 HTML + JavaScript，引入 **ECharts 5**（CDN），无需 Node/打包
- K 线图：`candlestick` 类型，下方 grid 叠加成交量 `bar`；默认 `dataZoom` 只显示最近 ~60–80 根，可滚轮/拖动看更早
- RSI 图：独立 `<div>`，与 K 线图通过 `echarts.connect()` 联动十字线，**共用同一 `dataZoom`**
- 阈值虚线：RSI 图中用 `markLine` 绘制，标注阈值数字
- **触发价横线**：K 线图中用水平 `markLine` 画各触发价（下穿绿 / 上穿红）；hover 预警表行时高亮对应横线
- **状态条**：图上方一行展示「当前 RSI 值 + 距最近阈值还需涨跌幅 + 下一交易日日期」
- 预警表：原生 `<table>` + JS 动态渲染，涨跌幅颜色：涨红 `#e84040`，跌绿 `#2dc08e`
- **预警表三态渲染**：正常价格 / `已触及 ✓`（置灰）/ `需超涨跌停`（置灰，不显示假价格）
- **市场自动推断**：默认按代码前缀推断 SH/SZ/BJ（6/5/9→SH，4/8/920→BJ，其余→SZ），可手动覆盖
- **查询 vs 强制更新分开**：代码框回车 / 「查询」走普通查询；「强制更新」调 `/api/refresh`
- **参数持久化**：RSI 周期、阈值列表存 `localStorage`
- **补齐三态**：loading（拉取中）、空数据 / 代码不存在、网络或接口错误，都要有提示，不能白屏
- **不要把企业微信 webhook key 带进来**：原脚本 `calc_rsi_trigger_price.py` 里硬编码了真实 webhook key，
  复用其代码时务必去掉密钥（本页面无推送需求，可直接删除推送相关逻辑）

## 七、数据缓存策略

**存储方案不需要优化**：单品种数据量极小（300 根日线 ≈ 20–40 KB），读写模式简单（每天每品种最多覆盖写一次、读多次），
「一文件一缓存」的 JSON 方案完全够用，瓶颈始终在 opentdx 网络拉取，不在存储。**不要上 SQLite/数据库**，属于过度设计。

- 文件路径：`data/{market}_{code}.json`（`market` 用 `SH`/`SZ`/`BJ`）
- 结构：`{"fetched_at": "2026-06-22T11:05:00", "session": "intraday|closed", "bars": [...]}`
- 判断逻辑（**需区分盘中 / 盘后**）：
  - 盘中（09:30–15:00）拉到的数据已去掉实时最新一根，标记 `session=intraday`，仅在当日盘中有效；
  - 盘后拉到完整收盘（含当日终值），标记 `session=closed`。
  - 命中条件：文件存在 + `fetched_at` 为今日 + 不会出现「盘后却读到盘中残缺缓存」的情况
    （即：盘后读取时，若缓存是 `intraday` 则视为失效、重新拉取）。
  - 否则 → 调 opentdx 重新拉取，覆盖缓存。
- 手动刷新接口（`/api/refresh`）：强制忽略缓存，重新拉取。

**写入要原子化**：先写 `data/{market}_{code}.json.tmp`，再 `os.replace()` 重命名覆盖（同盘 rename 原子），
避免写到一半进程中断导致缓存文件损坏、下次读取崩溃。

**保持「每天整体覆盖」，不要做增量追加**：`bars` 存的是**前复权**价，前复权价会随新除权整体平移，
昨天缓存的历史前复权值今天可能变化。每天整段重拉覆盖才正确；若为省网络改成只追加新 bar，历史段会算错。

## 十二、已定决策（原开放问题）

- **交易日历**：用 `akshare.tool_trade_date_hist_sina()` 获取，落地到本地缓存（如 `data/trade_calendar.json`）。
  调用时若**当前日期不在缓存覆盖区间**，则重新调 `akshare.tool_trade_date_hist_sina()` 更新缓存。
  「下一交易日日期」= 在该日历中取今日之后的第一个交易日（自动跳周末与法定节假日）。
  → 依赖新增 `akshare`（见第三节）。
- **期货**：v1 **不支持**，只做沪深 A 股（含北交所）。第五节涨跌停按代码前缀判断板块（±10/20/30/5%），不含期货。
- **前复权基准**：opentdx 的 QFQ 以「最新」为基准前复权。基准随新除权变化，已由第七节「每日整体覆盖、不增量追加」覆盖此问题。
