# 检查小主机上是否已跑过 Cloudflare Tunnel

如果之前在小主机（Linux）上跑过 `cloudflared`，按「服务 → 隧道 → 配置 → DNS」
逐层检查，判断是直接复用还是重建。

## 1. cloudflared 是否装了、版本多少

```bash
which cloudflared && cloudflared --version
```

## 2. 是否作为服务在跑

```bash
systemctl status cloudflared                  # active (running) 说明在跑
journalctl -u cloudflared -n 50 --no-pager    # 最近日志，看有没有报错 / 连上边缘
```

没有这个服务名时，也看看进程：

```bash
ps aux | grep -i cloudflared | grep -v grep
```

## 3. 本机建过哪些隧道

```bash
cloudflared tunnel list
```

列出隧道名、**TUNNEL_ID**、创建时间、连接数（CONNECTIONS）。有连接数说明当前正连着 Cloudflare。

查某条隧道的实时连接：

```bash
cloudflared tunnel info <隧道名或ID>
```

## 4. 凭证和配置文件在哪

```bash
ls -la ~/.cloudflared/            # 用 root 跑的话看 /root/.cloudflared/ 和 /etc/cloudflared/
cat ~/.cloudflared/config.yml     # 看 ingress 映射到哪个本地端口、绑了哪个域名
```

重点看 `config.yml` 里的：

- `tunnel:` / `credentials-file:` —— 用的是哪条隧道
- `ingress:` 的 `hostname` 和 `service:`（比如是不是 `http://127.0.0.1:8000`）

`<TUNNEL_ID>.json` 就是凭证文件，在则说明 `tunnel login` + `create` 都做过了。

## 5. 域名侧的解析是否还指向隧道

```bash
cloudflared tunnel route dns <隧道名> <你的域名>   # 幂等，已存在会提示 already exists
```

或在 Cloudflare 后台 DNS 看那条记录是不是 `CNAME → <TUNNEL_ID>.cfargotunnel.com`（橙云代理）。

## 6. 端到端验证

```bash
curl -s 127.0.0.1:8000 | head     # 本地 app 是否在响应
curl -I https://你的域名           # 走公网 + 隧道是否通（看到 cf-ray 头说明经过了 Cloudflare）
```

---

## 快速判断

| 现象 | 结论 |
|------|------|
| 服务 running + `tunnel list` 有连接数 + `curl 域名` 通 | 隧道完好，**直接复用**；第 5 步 `route dns` 确认域名即可 |
| 服务存在但 inactive / 报错 | `journalctl` 看原因（常见：凭证过期、`config.yml` 端口写错、app 没起） |
| 没有 `~/.cloudflared/`、`tunnel list` 为空 | 当时只是临时跑过 `cloudflared tunnel --url`（快速隧道），无持久配置，按 [`部署指南.md`](部署指南.md) 第 3 步正式建一条 |

---

## 已有隧道时，还要不要再建一条？

一台小主机上建多条隧道没问题，但**大多数情况不需要**——给现有隧道**加一条 ingress 规则**就够了。

### 方式 A：复用现有隧道，加一个 hostname（推荐）

一条隧道可同时映射多个域名/服务。在现有 `~/.cloudflared/config.yml` 的 `ingress` 里加一段：

```yaml
tunnel: <现有的 TUNNEL_ID>
credentials-file: /home/ubuntu/.cloudflared/<TUNNEL_ID>.json

ingress:
  - hostname: rsi.example.com        # 新加：给 rsi-alert 用
    service: http://127.0.0.1:8000
  - hostname: 别的.example.com        # 你原来已有的服务
    service: http://127.0.0.1:9000
  - service: http_status:404         # 兜底必须放最后
```

再给新域名加路由并重启：

```bash
cloudflared tunnel route dns <现有隧道名> rsi.example.com
sudo systemctl restart cloudflared
```

**优点**：一个 cloudflared 服务管所有站点，省资源、好维护。`ingress` 按顺序匹配，
`http_status:404` 兜底规则**必须在最后**。

### 方式 B：再建一条独立隧道

```bash
cloudflared tunnel create rsi-alert
```

也能跑，但注意：

- 每条隧道有**独立的 config 和凭证**，`cloudflared service install` 默认只跑**一个** config。
  要同时跑两条独立隧道，得手动配**两个不同名的 systemd 服务**，分别 `--config` 指向各自的 yml。
- 适合**完全隔离**的场景：不同访问策略、不同 Cloudflare 账号 / Zero Trust 配置，或想随时单独停掉其中一个。

### 怎么选

- 已有隧道正常 → 用**方式 A** 加 `rsi.example.com` 这条 ingress，最省事。
- 想让 rsi-alert 与原服务彻底分开管理 → 才用**方式 B** 另配独立服务。
